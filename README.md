金融投资助手（Financial Investment Agent）
目标定位：一个达到企业级应用标准的、基于 LangChain 的智能投资助手。系统以 Agent 为核心，通过自然语言交互完成资产查询、市场分析、策略生成与交易执行，并在架构、安全与工程规范上贴近真实金融系统。

1. 项目背景与目标
1.1 项目背景
随着大模型与 Agent 技术的发展，金融领域逐渐从“工具驱动”向“智能决策辅助”转变。本项目旨在探索 LLM + Agent + 金融系统 的工程化落地方式，而不仅是 Demo 或脚本级应用。
1.2 项目目标
构建一个 可通过聊天框交互 的金融投资 Agent
用户可使用自然语言完成资产管理与投资操作
项目在以下方面满足企业级标准：
o清晰的项目结构
o模块化、可扩展架构
o规范的配置管理
o严格的安全与权限控制
o可测试、可部署、可审计

2. 核心需求分析
2.1 功能性需求
2.1.1 资产管理（Asset Management）
通过对话形式支持以下能力：
查看当前投资资产整体状况
按金融产品类型统计资产（如：股票 / 基金 / 加密货币 / 现金等）
单个资产的关键信息：
o持有数量
o成本价 / 当前市场价
o浮动盈亏、涨跌幅
o仓位占比
⚠️ 注意：Agent 不直接持有资产数据，而是通过 Tool 调用受控的数据访问层。

2.1.2 市场信息查看（Market Overview）
支持用户通过自然语言查询市场概况
初期方案：
oAgent 返回结构化摘要 + 外部网页链接（如行情网站）
后期可扩展：
o直接拉取行情数据并生成总结

2.1.3 投资策略创建与分析（Strategy & Analysis）
用户可通过对话创建或修改投资策略，例如：
o“帮我基于 MA 和 RSI 生成一个短线策略”
o“回测最近 6 个月该策略的表现”
系统能力要求：
o实时或准实时获取 K 线数据
o支持常见技术指标计算（MA / EMA / RSI / MACD / Bollinger Bands 等）
o策略逻辑可结构化存储（非纯 Prompt）

2.1.4 交易执行（Order Execution）
Agent 可根据：
o用户明确指令
o策略信号 触发下单流程
下单流程必须满足：
o明确意图确认（二次确认 / 风险提示）
o严格权限校验
o可审计的交易日志
❗ 任何交易行为必须是“可追溯、可回滚（模拟环境）”的

2.2 非功能性需求
2.2.1 安全性（重点）
模拟真实金融系统的安全要求：
oAPI Key / Token 加密存储
o不允许 LLM 直接接触敏感凭证
oTool 层权限分级（只读 / 可交易）
防御风险：
oPrompt Injection
o越权调用 Tool
o幻觉导致的错误下单

2.2.2 工程化要求
清晰的分层架构（Agent ≠ 业务 ≠ 基础设施）
配置与代码解耦
支持：
o单元测试
o集成测试
oMock 外部金融接口

4. 技术选型
4.1 Agent 框架
LangChain
oReAct / Plan-and-Execute Agent
oTool Calling
oMemory（短期 + 长期）
MCP（Model Context Protocol）
o统一上下文注入标准
o控制模型可见信息边界

4.2 数据与计算
行情数据：
o第三方行情 API（模拟 / 实盘）
指标计算：
opandas / numpy / ta-lib（或自实现）

5. 安全设计（关键设计点）
5.1 Agent 安全边界原则
LLM 永远不直接执行真实世界副作用
所有高风险行为：
o必须通过 Tool
o必须校验上下文状态
5.2 下单安全策略
明确区分：
o模拟交易环境（Paper Trading）
o实盘交易环境
强制流程：
1.Agent 生成交易意图
2.风控模块校验
3.用户确认
4.执行并记录审计日志

6. 后续可扩展方向
多 Agent 协作（分析 Agent / 执行 Agent / 风控 Agent）
策略版本管理与评估
用户画像与风险偏好建模
企业级部署（Docker / K8s / CI/CD）

7. 总结
该项目不是一个简单的“聊天 + 金融 API”应用，而是：
一个以 Agent 为核心、以安全与工程规范为前提的智能金融系统原型。
后续文档建议进一步拆分为： - 项目目录结构设计 - Tool 设计规范 - Agent Prompt 与策略规范 - 安全与风控设计文档
